## 一、立题依据

### （一）研究目的

随着"健康中国2030"战略的深入推进，国民健康意识不断增强，个性化健康管理需求日益凸显。中医体质学说作为中医药理论的重要组成部分，在疾病预防、健康管理和养生保健方面具有独特优势。然而，传统中医体质辨识和膳食调理主要依赖人工经验，存在主观性强、标准化程度低、个性化服务不足等问题。

当前，人工智能、大数据等现代信息技术在医疗健康领域得到广泛应用，但将中医体质理论与智能算法深度融合的膳食推荐系统仍属空白。现有健康管理平台多侧重于西医指标监测，缺乏基于中医理论的个性化膳食指导功能，难以满足人民群众对中医养生服务的迫切需求。

本课题旨在运用Vue.js、Node.js、MongoDB等现代技术栈，结合DeepSeek人工智能模型，设计并实现一个基于中医体质理论的智能膳食推荐系统。通过信息化手段构建集体质辨识、智能推荐、AI对话、菜谱管理于一体的综合平台，实现从体质分析到个性化膳食推荐的全流程智能化服务，为用户提供科学、精准、便捷的中医膳食指导，推动中医药服务的数字化转型。

### （二）研究意义

1. **理论意义：** 本课题将传统中医体质学说与现代人工智能技术深度融合，探索"互联网+中医药"的创新发展模式。通过构建标准化的体质辨识模型和个性化的膳食推荐算法，为中医体质理论的现代化研究提供技术支撑，推动中医药理论的创新发展。

2. **实践意义：** 系统将为用户提供科学、个性化的膳食指导服务，帮助用户根据自身体质特点选择合适的饮食方案，提高健康管理的针对性和有效性。同时，通过AI营养师功能，降低中医养生服务的门槛，让更多人享受到专业的中医膳食指导。

3. **技术意义：** 采用前后端分离架构，集成Vue.js、Node.js、MongoDB、OpenAI等先进技术，构建模块化、可扩展的系统平台。通过机器学习算法实现智能推荐，为同类健康管理系统的开发提供技术参考和实现范例。

4. **社会意义：** 响应"健康中国2030"战略号召，推动中医药服务数字化转型，提升全民健康素养。系统有助于传承和发展中医药文化，促进中医药现代化、国际化发展。

### （三）国内外研究进展和趋势

#### 1. 国内研究现状及发展趋势

国内在中医体质学说研究方面已形成较为完善的理论体系。王琦等学者提出的九种体质分类法得到广泛认可，为体质辨识标准化奠定了基础。钱会南等学者深入研究了饮食五味对体质的影响，指出饮食营养与体质形成密切相关，为体质与膳食调理的结合提供了理论依据。

近年来，随着信息技术的发展，国内开始探索中医与现代技术的结合。基于SpringBoot、Vue.js等框架的健康管理系统不断涌现，但专门针对中医膳食推荐的智能化平台相对较少。部分研究尝试将机器学习算法应用于体质辨识，但多局限于单一功能模块，缺乏系统性的解决方案。

随着人工智能技术的发展，国内开始探索AI在中医领域的应用。基于深度学习的中医问诊系统、智能舌诊平台等相继出现，但将AI技术与中医体质理论深度融合的膳食推荐系统仍处于起步阶段。

政策层面，《国民营养计划（2017-2030年）》和《健康中国2030规划纲要》明确提出要发展传统食养服务，推广中医食养理念，为中医药与现代信息技术结合提供了政策支持。

总体来看，国内研究呈现标准化、技术融合、个性化、政策支持和国际化五大发展趋势，中医体质分类正向标准化方向发展，人工智能技术正在与中医药深度融合。

## 二、主要研究内容

本课题的核心研究内容是围绕中医体质理论与现代信息技术的深度融合，设计并实现一个智能化的膳食推荐系统。系统将采用前后端分离架构，后端基于Node.js + Koa框架构建RESTful API，前端采用Vue.js + Element Plus开发用户界面，数据库使用MongoDB进行数据存储，集成DeepSeek AI模型实现智能对话和推荐功能。

### （一）体质辨识模块研究

**1. 理论基础构建**

- 深入研究王琦院士提出的九种体质分类理论（平和质、气虚质、阳虚质、阴虚质、痰湿质、湿热质、血瘀质、气郁质、特禀质）
- 分析每种体质的生理特征、心理特点、发病倾向和调理原则
- 建立体质与食材性味、归经、功效的对应关系数据库

**2. 问卷量表设计**

- 设计包含60个问题的标准化体质辨识问卷，涵盖形体特征、生理表现、心理特征、适应能力等多个维度
- 采用5级李克特量表进行量化评分，确保辨识结果的客观性和准确性
- 建立问题权重体系，不同问题对体质判定的影响程度进行差异化设置

**3. 智能辨识算法实现**

- 开发基于模糊数学理论的体质判定算法，处理用户回答的模糊性和不确定性
- 实现兼夹体质识别功能，支持用户同时具有多种体质特征的复杂情况
- 设计体质变化追踪机制，记录用户不同时期的体质状态变化趋势

### （二）智能推荐算法研究

**1. 多维度匹配算法设计**

本系统的核心在于智能推荐算法，采用四层匹配架构来实现精准推荐：

- **体质匹配层**：这是最关键的一层，系统会根据菜谱的性味特点（寒、凉、平、温、热）和归经属性，结合用户的体质类型来计算基础匹配度。比如说，阳虚体质的用户更适合温补类的食物，系统会优先推荐性温的菜谱。

- **偏好匹配层**：这一层主要考虑用户的口味偏好，分析用户对五味（酸、甜、苦、辣、咸）的喜好程度。如果用户特别喜欢甜味，系统就会在推荐中适当提高甜味菜谱的权重。

- **禁忌过滤层**：为了保障用户安全，系统会自动识别并过滤掉不适合的菜谱。比如用户对某些食材过敏，或者有宗教饮食禁忌，系统都会智能排除这些选项。

- **场景适配层**：考虑到用户的实际使用场景，系统会综合用餐时间、烹饪难度等因素进行优化。比如早餐时间会推荐清淡易消化的食物，晚餐则会推荐营养丰富的主食类菜谱。

**2. 推荐算法实现方案**

系统的推荐算法采用多因子综合评分机制，主要考虑以下几个维度：

首先是体质匹配度，这是评分的核心依据。系统会根据中医理论的"性味归经"原则，判断菜谱是否适合用户的体质特点。比如阴虚体质的用户，系统会优先推荐性凉、具有滋阴功效的菜品。

其次是口味偏好匹配，系统会分析用户对不同味道的喜好程度。通过收集用户的历史行为数据，建立个人口味画像，实现更精准的口味推荐。

再者是饮食禁忌检查，这是保障用户健康的重要环节。系统会严格检查菜谱中是否含有用户禁忌的食材，一旦发现会大幅降低推荐分数。

最后是场景适配度，系统会根据当前的用餐场景、时间限制、烹饪条件等因素，调整推荐策略。比如用户设置了30分钟的烹饪时间限制，系统就会优先推荐制作简单的快手菜。

**3. 个性化优化策略**

为了让推荐更加精准，系统采用了多种个性化优化策略：

通过协同过滤技术，系统会分析具有相似体质和偏好的用户群体，挖掘他们的共同喜好，从而为新用户提供更准确的推荐。比如发现大部分气虚体质的用户都喜欢某种类型的汤品，系统就会向其他气虚体质用户推荐类似的菜品。

基于内容的推荐算法会深入分析菜谱的特征属性，包括食材搭配、烹饪方法、营养价值等，建立菜品之间的关联关系。这样即使用户没有明确的历史行为，系统也能根据菜品的内在特征进行合理推荐。

系统还会引入时间衰减因子，动态调整历史偏好的权重。用户的口味可能会随着时间变化，系统会根据行为的时间远近，给予不同的权重，确保推荐结果能够反映用户当前的真实需求。

针对新用户的冷启动问题，系统设计了智能的默认推荐策略。通过分析用户的初始设置信息，结合大众偏好数据，为新用户提供合理的初始推荐，并在用户交互过程中不断学习和优化。

### （三）AI对话系统研究

**1. 知识库构建**

- 构建包含中医基础理论、体质学说、食疗养生、食材功效等内容的结构化知识库
- 整理常见健康问题、养生咨询、膳食建议等对话场景模板
- 建立专业术语词典，确保AI回复的准确性和专业性

**2. 大语言模型集成**

- 集成DeepSeek大语言模型，配置专业的系统提示词（System Prompt）
- 设计角色设定：AI作为"专业的中医体质健康顾问"，具备丰富的中医理论知识和临床经验
- 实现流式响应机制，支持打字机效果，提升用户体验

**3. 对话管理实现**

在对话管理方面，系统设计了完整的智能对话架构。首先为AI助手设定了专业的角色定位，让其扮演"养生小助手"这一亲切的中医体质健康顾问形象。系统为AI配置了丰富的专业知识库，包括中医九种体质的分类方法、不同体质的调理原则、食材的性味归经理论、四季养生的基本理念，以及常见亚健康状态的中医调理方案。

为了确保AI回复的专业性和安全性，系统制定了详细的回答规范。AI需要使用通俗易懂的语言与用户交流，避免过多的专业术语，让普通用户也能轻松理解。在给出建议时，AI会结合用户的具体体质特点，提供个性化的健康指导。同时，系统设置了安全边界，当遇到严重健康问题时，AI会建议用户及时就医检查，避免做出绝对的医疗诊断，确保用户安全。

在技术实现上，系统采用了先进的流式响应技术，支持实时的打字机效果，让用户能够感受到与真人对话的体验。同时，系统还实现了多轮对话的上下文管理，能够记住之前的对话内容，保持话题的连贯性，提供更加自然流畅的交互体验。

### （四）系统架构与技术实现

**1. 整体架构设计**

- **前端架构**：采用Vue 3 + Composition API，配合Element Plus UI组件库，实现响应式界面设计
- **后端架构**：基于Node.js + Koa2框架，采用RESTful API设计模式，支持高并发访问
- **数据库设计**：使用MongoDB存储用户信息、体质数据、菜谱信息等，Redis用于缓存热点数据
- **AI服务**：集成OpenAI API，实现智能对话和菜谱生成功能

**2. 核心功能模块实现**

**用户管理模块：**

在用户管理方面，系统设计了完整的用户信息管理体系。每个用户都有唯一的用户名作为身份标识，系统会记录用户的基本信息。最重要的是用户的体质信息，系统支持九种体质类型的记录，包括平和质、气虚质、阳虚质等，用户可以通过体质测试来确定自己的体质类型。

系统还详细记录了用户的个人偏好设置，包括对五种基本味道（酸、甜、苦、辣、咸）的喜好程度评分，用户的饮食禁忌清单，过敏原信息，以及烹饪相关的偏好（如最大烹饪时间、烹饪难度等）。这些数据为个性化推荐提供了重要依据。

此外，系统还保存用户的会话数据，包括与AI的对话历史记录，以及用户收藏的菜谱信息。这样用户可以随时查看之前的咨询记录，方便回顾和继续对话，同时也能快速找到自己喜爱的菜谱。

**菜谱管理模块：**

在菜谱管理方面，系统建立了完整的菜谱信息数据库。每道菜谱都包含基本信息，如菜名、详细描述等核心内容。系统特别注重中医属性的记录，包括菜谱的性味特点（寒、凉、平、温、热），味道特征（酸、甘、苦、辛、咸），以及归经信息，这些都是中医食疗理论的重要组成部分。

系统还会标注每道菜谱适合的体质类型和不适宜的体质类型，为个性化推荐提供重要依据。在食材信息方面，系统详细记录所需食材的名称、用量，并配有相应的图标，让用户能够直观地了解所需材料。烹饪步骤会按照顺序详细说明，确保用户能够按部就班地完成制作。

此外，系统还记录烹饪时间、难度等级等实用信息，并支持标签分类功能，方便用户根据特定需求进行筛选。为了关注用户的营养健康，系统还提供了营养成分数据，包括热量、蛋白质、脂肪、碳水化合物等关键指标，帮助用户更好地控制饮食健康。

**推荐算法模块：**

推荐服务是系统的核心功能，主要负责为用户提供个性化的菜谱推荐。系统首先会获取用户的完整信息，包括体质类型和个人偏好设置，然后从活跃的菜谱库中筛选出合适的菜品。

在推荐过程中，系统会为每道菜谱计算个性化的匹配分数。这个分数综合考虑了多个维度：首先是体质匹配度，系统会根据用户的体质特点，判断菜谱是否适合；其次是口味偏好匹配，分析用户对不同味道的喜好；再者是饮食禁忌检查，确保推荐的菜谱不会包含用户忌讳的食材；最后是场景适配，考虑用户的用餐时间、烹饪条件等实际因素。

系统采用加权评分机制，为不同因素分配合理的权重，确保推荐结果既符合中医理论，又满足用户的个性化需求。最终，系统会按照匹配分数从高到低进行排序，为用户推荐最适合的菜品，并且支持限制推荐数量，避免信息过载。

**前端界面实现：**

**响应式布局设计：**

**安全措施实现：**

在数据安全方面，系统实施了多层次的安全保护机制。首先是用户身份认证，系统采用令牌认证方式，用户在登录后会获得一个安全的访问令牌，后续的每次请求都需要携带这个令牌，系统会验证令牌的有效性，确保只有合法用户才能访问系统资源。

对于敏感数据的存储和传输，系统采用了先进的加密技术。用户的个人信息、健康数据等重要内容都会进行加密处理，即使数据被非法获取，也无法直接读取其中的内容。系统还实现了完善的输入验证机制，对用户提交的所有数据进行严格的格式检查和内容过滤，防止恶意代码注入，确保系统的安全稳定运行。

**缓存机制实现：**

为了提升系统响应速度和用户体验，系统设计了智能的缓存机制。当用户请求推荐结果时，系统首先会检查缓存中是否已经有相应的结果，如果有则直接返回，避免重复的数据库查询和计算，大大缩短了响应时间。

缓存数据设置了合理的有效期，推荐结果通常缓存一小时，这样既保证了数据的实时性，又避免了频繁的重复计算。当用户的偏好设置发生变化时，系统会自动清除相关的缓存数据，确保用户能够看到最新的推荐结果。

系统还实现了分布式的缓存管理，支持大规模用户的并发访问。通过智能的缓存失效策略，系统能够在数据更新时及时清理过期缓存，保证用户始终获得准确的信息。

数据库性能是系统高效运行的关键，系统从多个方面进行了优化。首先是索引优化，针对常用的查询条件建立了复合索引，比如在菜谱的适合体质和性味字段上建立联合索引，大大加快了查询速度。

查询语句的优化也是重点，系统采用了聚合查询等技术，能够在单次数据库操作中就完成复杂的数据处理任务，减少了数据库的访问次数。同时，系统还实现了查询结果的分页处理，避免一次性返回过多数据，减轻数据库和服务器的压力。

在数据存储方面，系统采用了合理的数据结构设计，将相关的数据进行有效的组织和关联，既保证了数据的完整性，又提高了查询效率。通过这些优化措施，系统能够在高并发情况下依然保持稳定的性能表现。

```javascript
// 推荐服务实现
class RecommendationService {
	async getPersonalizedRecommendations(userId, options = {}) {
		const user = await User.findById(userId).populate("preferences");
		const recipes = await Recipe.find({ isActive: true });

		// 计算每个菜谱的匹配度
		const scoredRecipes = recipes.map((recipe) => {
			const score = this.calculateMatchScore(recipe, user);
			return { ...recipe.toObject(), matchScore: score };
		});

		// 按匹配度排序并返回
		return scoredRecipes
			.sort((a, b) => b.matchScore - a.matchScore)
			.slice(0, options.limit || 10);
	}

	calculateMatchScore(recipe, user) {
		let score = 80; // 基础分数

		// 体质匹配（权重40%）
		score += this.calculateConstitutionScore(recipe, user.constitution) * 0.4;

		// 口味偏好（权重25%）
		score +=
			this.calculateFlavorScore(recipe, user.preferences.flavorPreference) *
			0.25;

		// 禁忌检查（权重20%）
		score += this.checkRestrictions(recipe, user.preferences) * 0.2;

		// 场景适配（权重15%）
		score += this.calculateScenarioScore(recipe, user.preferences) * 0.15;

		return Math.round(Math.max(50, Math.min(99, score)));
	}
}
```

```javascript
// Redis缓存策略
class CacheService {
	async getRecommendations(userId) {
		const cacheKey = `recommendations:${userId}`;
		const cached = await redis.get(cacheKey);

		if (cached) {
			return JSON.parse(cached);
		}

		// 从数据库获取并缓存
		const recommendations = await this.generateRecommendations(userId);
		await redis.setex(cacheKey, 3600, JSON.stringify(recommendations)); // 1小时缓存

		return recommendations;
	}

	async invalidateUserCache(userId) {
		const patterns = [`user:${userId}:*`, `recommendations:${userId}`];
		for (const pattern of patterns) {
			const keys = await redis.keys(pattern);
			if (keys.length > 0) {
				await redis.del(...keys);
			}
		}
	}
}
```
